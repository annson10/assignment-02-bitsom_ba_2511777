## Vector DB Use Case  
A traditional keyword-based database search would not be sufficient for this use case. Keyword search relies on exact word matching and does not understand the semantic meaning of queries. In legal documents, the same concept can be expressed using different terminology. For example, a query like “termination clauses” may not return relevant sections if the document uses phrases such as “contract cancellation conditions” or “agreement expiry terms.” This limitation makes keyword search unreliable for large, complex documents like 500-page contracts. It also struggles with natural-language questions because it does not reliably understand intent, synonyms, or context.  

A vector database would solve this by supporting semantic search. In this approach, contract sections and the user’s question are converted into embeddings, which are numerical vectors that capture the contextual meaning rather than just exact words. When a lawyer asks a question in plain English, the system converts the query into a vector and can then retrieve similar sections of the contract whose embeddings are closest to the query embedding, even if exact keywords are not present.  

A vector database plays a crucial role by efficiently storing and indexing these embeddings, allowing fast similarity searches across large volumes of text. It can return the most relevant passages ranked by similarity, significantly improving search accuracy and user experience. In practice, I would recommend a hybrid design rather than replacing keyword search entirely. Keyword search can still help to a small extent with exact citations, defined terms, section numbers, or party names, while the vector database handles meaning-based retrieval across long contracts.  

Therefore, while keyword search may serve as a basic filter, a vector database is essential for building an intelligent, scalable, and user-friendly contract search system that aligns with how humans naturally ask questions.
